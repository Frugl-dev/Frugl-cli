# Implementation Plan: poppi-cli org membership — `poppi join` + org-aware `whoami` / `upload`

**Branch**: `004-cli-org-join` | **Date**: 2026-05-24 | **Spec**: [./spec.md](./spec.md)

**Input**: Feature specification from `/specs/004-cli-org-join/spec.md`

## Summary

This feature is the **client side** of the cloud's org-membership work (`poppi/specs/003-org-membership-permissions`). It adds **one** new subcommand — `poppi join <code>`, which redeems a base32-Crockford invite code against the cloud's `POST /api/join` and adds the authenticated user to an Organization at the role baked into the code — and makes **two** existing commands org-aware: `poppi whoami` reports the caller's Organization + role, and `poppi upload` resolves the destination Organization up front (gating brand-new no-Membership users before any anonymization, and naming the destination in its pre-upload summary).

The technical approach is **pure reuse** of the foundation laid by `001-cli-ingest-client`: the same native-`fetch` HTTP client with its CLI-version header and bounded-retry predicate, the same `@napi-rs/keyring` token read, the same `zod` cloud-response validation, the same version-gate, the same `--json` output-mode plumbing, and the same exit-code table. No new runtime dependencies are introduced (spec Assumptions → "No new dependencies"). The CLI never sends an `org_id` — the cloud derives it from the caller's single Membership (cloud FR-023) — so this feature adds **no new persistent local state**: org context is re-resolved every run from `GET /api/orgs/me`.

The two consumed endpoints both layer onto the existing cloud HTTP boundary (`contracts/cloud-api.md`):

- `POST /api/join` — redeem a code; success returns `{ org, membership }`; typed errors map 1:1 to user-facing messages and exit codes.
- `GET /api/orgs/me` — resolve the caller's Organization + role; `409 org_required` is the "no org yet" signal (a reported state for `whoami`, a hard gate for `upload`).

The work adds **four** new exit codes to the public FR-037 contract (`ORG_REQUIRED`, `JOIN_CODE_REJECTED`, `ALREADY_IN_OTHER_ORG`, `RATE_LIMITED`) within the reserved gaps, never reassigning an existing code.

## Technical Context

**Language/Version**: TypeScript 6.x, Node.js ≥ 20 (unchanged from `001`).

**Primary Dependencies**: No additions. This feature consumes the dependency set already locked by `001-cli-ingest-client`:

- CLI framework: **oclif** (`@oclif/core`) — `poppi join` is a new Command class auto-discovered via `src/index.ts`.
- Cloud-contract validation: **`zod`** — new schemas for `/api/join` and `/api/orgs/me` responses in `src/cloud/schemas.ts`.
- Version-gate parsing: **`semver`** — reused unchanged for the shared `426` behaviour.
- Bounded retry: **`p-retry`** — reused via `src/lib/retry.ts`; the same transient-only predicate applies.
- Bounded concurrency: **`p-limit`** — not exercised by this feature (no per-session fan-out in `join`/`orgs/me`).
- Interactive prompts: **`@inquirer/prompts`** — not exercised (no prompt in `join`; `upload`'s existing confirm prompt is reused, gaining one org line).
- Cross-platform state-dir persistence: **`conf`** — not exercised (no new persistent state, per spec Assumptions).
- OS credential store: **`@napi-rs/keyring`** — reused via `src/auth/keychain.ts` (read-only here).
- HTTP client: Node native `fetch` via `src/cloud/client.ts` — reused unchanged.
- Color: **`picocolors`** — reused for `join`/`whoami` output (respect `NO_COLOR`/`--no-color`, FR-022).

**Storage**: None added. `poppi join`, org-aware `whoami`, and the upload org-gate add nothing to the OS keychain, the `poppi-ledger` `conf` namespace, or the `poppi-resume-state` `conf` namespace. Org context is ephemeral, re-resolved per invocation (spec Key Entities → "Org context (CLI-side, ephemeral)"; spec Assumptions → "No new persistent local state"). The auth token is **read** from the keychain exactly as `001` does; `poppi login` remains the only writer.

**Testing**: vitest, co-located `*.test.ts` (matches `001`). Three tiers, as in `001`:

- **Unit** — base32-Crockford normalization/validation (FR-004/005), the `409 already_member` → exit-0 mapping (FR-017), `Retry-After` parsing (FR-014), no-plaintext-code-leakage (SC-005).
- **Contract** — `zod` round-trip the `/api/join` success + every typed-error body and the `/api/orgs/me` success + `409 org_required` body against the recorded schemas in `contracts/`. Drift = test failure (the cross-repo drift sentinel, Principle VI).
- **Integration** — `join` → `whoami` → `upload` against the local Docker stack brought up from the sibling `poppi/` repo (`pnpm stack:up`), using a code generated by an Admin on the dashboard. Mock-server tests assert the no-auth fast-fail makes zero network calls (SC-002) and the `ORG_REQUIRED` gate hits no upload endpoint (SC-008).

**Target Platform**: Cross-platform — macOS, Windows, Linux on Node ≥ 20 (unchanged from `001`).

**Project Type**: Single-package CLI. No structural change; new files slot into the existing `src/` layout established by `001`.

**Performance Goals**:

- **SC-001 (join time-to-value)**: code-in-clipboard → joined-and-authorised in **< 30 s** with one shell command, against the local stack. `POST /api/join`'s server-side p95 ≤ 500 ms (cloud `join.md`) leaves ample headroom; the CLI adds only normalization + one round-trip.
- **Org-resolution overhead on `upload`**: one `GET /api/orgs/me` call before discovery — a single control-plane round-trip on the existing 8 s timeout budget (`001` R-3). It runs **before** the SC-003a classification budget, so it does not eat into it.

**Constraints**:

- **Privacy posture (Principle VI; `001` FR-034/035)**: no telemetry, no background activity. Every `GET /api/orgs/me` and `POST /api/join` is a direct, traceable consequence of a user command.
- **No-plaintext-code-leakage (FR-006, SC-005)**: the invite code grants org access and is a secret; it MUST NOT appear in default-level output. Only `--debug` may include it, with a docs warning.
- **Fail-closed on contract drift (FR-012, SC-006)**: a success or error body that does not match the documented schema is a contract violation → `GENERIC_FAILURE`, never a silent partial success.
- **Org-gate before any local work (FR-027/028, SC-008)**: `GET /api/orgs/me` resolves the destination **before** discovery/anonymization; a no-Membership user transmits zero bytes and writes no inspection dir, even under `--dry-run`.
- **Retry policy (shared with `001` FR-029a/b)**: transient errors only (network reset, timeout, 5xx). All typed 4xx redemption errors, `401`, `426`, and `429` fail fast.

**Scale/Scope**:

- CLI surface delta: **+1** user-facing command (`join`), **2** existing commands extended (`whoami`, `upload`). No `poppi org` namespace (explicit out-of-scope).
- Endpoints consumed delta: **+2** (`POST /api/join`, `GET /api/orgs/me`).
- Exit-code delta: **+4** (`ORG_REQUIRED`, `JOIN_CODE_REJECTED`, `ALREADY_IN_OTHER_ORG`, `RATE_LIMITED`).
- One Organization per user in v1 (cloud assumption) — `GET /api/orgs/me` returns at most one org; `poppi upload` has exactly one destination. No `--org` flag, no org switching.

**Structure Decision**: Single-project layout, extending the `001` `src/` convention. The new code is split so that **transport** (`src/cloud/join.ts`, `src/cloud/orgs.ts`) is separate from **pure logic** (`src/join/normalize.ts`, `src/join/validate.ts`) and from the **command orchestration** (`src/commands/join.ts`). This mirrors `001`'s separation of `src/cloud/` (HTTP boundary) from `src/sources/`/`src/ledger/` (pure logic) from `src/commands/` (orchestrators), keeps the pure normalization/validation independently unit-testable (SC-005), and keeps the new zod schemas co-located with the existing ones in `src/cloud/schemas.ts` so the contract-drift sentinel stays in one place. `poppi join` lives directly under `src/commands/` (top-level verb, FR-001) rather than a `join/` subgroup, matching oclif auto-discovery and the existing `login`/`logout`/`whoami`/`upload` siblings. No `poppi org` namespace is introduced (explicit out-of-scope).

> **Note on the current scaffold state**: at the time of writing, `src/` still holds the pre-`001`-migration `commander`-based stubs (`src/commands/whoami.ts`, `src/commands/upload.ts`, `src/auth/keychain.ts` are stubs; there is no `src/cloud/` or `src/lib/` yet). This plan describes the layout **after** `001`'s Phase-1/2 migration to oclif lands, which is a hard prerequisite for this feature (it edits `001`-created files). The tasks.md sequencing makes that dependency explicit.
